# OpenClaw 文档架构设计

本页记录当前仓库采用的标准化文档结构，作为后续写作、迁移和站点化维护的基准。

## 设计目标

- 让新用户可以按学习路径逐步阅读。
- 让老用户可以按主题快速跳转。
- 让仓库结构和 MkDocs 导航保持一致，减少维护成本。
- 保留旧版教程内容，避免迁移过程中丢失历史资料。

## 顶层目录

```text
hand-on-openclaw/
├── README.md
├── LICENSE
├── mkdocs.yml
├── requirements.txt
├── docs/
├── docold/
└── examples/
```

## 目录职责

### `docs/`

新的主文档目录，所有后续教程内容都应优先维护在这里，并直接映射到 MkDocs 站点导航。

```text
docs/
├── index.md
├── install/
├── basic/
├── skills/
├── automation/
├── deep-dive/
└── projects/
```

各分区职责如下：

- `install/`：安装部署，面向首次接触 OpenClaw 的用户。
- `basic/`：基础使用，包括首次对话、模型选择、聊天渠道、Workspace 和 Memory。
- `skills/`：Skill 认知、安装、创建、发布和示例。
- `automation/`：定时任务、Heartbeat、Webhook 等自动化能力。
- `deep-dive/`：架构、源码走读、消息流和源码构建。
- `projects/`：面向结果的实战项目案例。

### `examples/`

示例资源目录，用于存放可直接参考或复用的配套文件。

```text
examples/
├── skills/
├── configs/
└── workflows/
```

- `skills/`：示例 Skill。
- `configs/`：示例配置和工作空间文件。
- `workflows/`：自动化工作流样例。

### `docold/`

旧版教程归档目录，完整保留原始拆分文档和截图资源。

- 用途是迁移参考，不作为新的主维护目录。
- 新增内容不再写入这里。
- 旧内容如果需要复用，应迁移到 `docs/` 后再继续更新。

## 信息架构

当前文档采用“从入门到进阶”的六层结构：

1. 安装部署：解决环境和启动问题。
2. 基础使用：解决“能不能跑起来并开始用”的问题。
3. Skills：解决“怎么扩展能力”的问题。
4. 自动化工作流：解决“怎么持续运行”的问题。
5. 源码解析：解决“内部怎么实现”的问题。
6. 实战项目：解决“如何落地到真实场景”的问题。

这个顺序同时兼顾了两类读者：

- 新手可以按顺序阅读。
- 有经验的用户可以直接进入某个专题目录。

## 与 MkDocs 的对应关系

仓库目录和站点导航保持一一对应，定义位于 `mkdocs.yml`：

- `docs/index.md` 对应站点首页。
- `docs/install/` 到 `docs/projects/` 对应一级导航栏目。
- 每个 Markdown 文件对应一篇独立页面，避免单文件过长。

这种组织方式的好处是：

- 文件路径稳定，便于协作引用。
- 页面职责明确，后续拆分或合并成本低。
- 可以平滑接入 GitHub Pages 或其他静态站点部署方式。

## 迁移原则

从 `docold/` 迁移到 `docs/` 时，建议遵循以下规则：

- 优先按主题归类，不机械保留旧编号。
- 截图、命令和报错信息尽量保留。
- 长章节拆成多个主题页，避免单页过大。
- 迁移后统一补齐目标、前置条件、步骤、验证结果和常见问题。

## 后续维护建议

- 新增文档时先确认它属于哪一个一级目录。
- 如果内容既是教程又是案例，优先放在 `projects/`，并在相关专题页交叉链接。
- 如果只是配置片段或模板，不放进 `docs/`，而是放进 `examples/`。
